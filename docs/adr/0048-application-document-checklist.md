# ADR-0048 — Application Document Checklist and Admissions Completion Workflow

**Date:** 2026-06-25
**Status:** Accepted
**Deciders:** Ricardo Julia (sole approver)
**Council review:** `docs/reviews/2026-06-25-council-review-10-partial-gap-closeout.md`

---

## Context

The admissions module supports staff review and decision workflows, and the `/apply` public portal
accepts applications. However, no mechanism exists for institutions to define what documents are required
for an application (transcripts, letters of recommendation, ID documents, pastoral reference letters),
for applicants to upload those documents, or for staff to track checklist completion before advancing an
application to a decision.

Faith-based institutions frequently require pastoral reference letters, baptism certificates, and
statement-of-faith documents that secular SIS platforms do not model. This checklist must be flexible
enough to support those institution-defined document types.

---

## Decision

### 1. Document type registry

Add `academy_document_types` table. Each row defines one document type an institution expects:

- `id`, `tenant_id`
- `name` — display label (e.g., "Pastoral Reference Letter")
- `slug` — machine identifier (e.g., `pastoral_reference`)
- `required` — boolean; if true, application cannot advance to decision without this document marked received or waived
- `description` — optional guidance shown to applicants
- `active` — boolean

Managed by admissions admins at `/admin/admissions/document-types`.

### 2. Application document records

Add `academy_application_documents` table:

- `id`, `tenant_id`, `application_id`
- `document_type_id` — references `academy_document_types`
- `status`: `pending` | `uploaded` | `received` | `waived`
- `storage_path` — Supabase Storage object path (null until uploaded)
- `uploaded_at`, `received_at`, `waived_at`
- `waived_by` — person_id of the staff member who waived the requirement
- `waiver_note` — required when status is `waived`
- `uploaded_by` — person_id (applicant token holder or staff)

### 3. Storage boundary

Documents are stored in a private Supabase Storage bucket `academy-application-documents`.
Paths follow the pattern: `{tenant_id}/{application_id}/{document_type_slug}/{uuid}.{ext}`.

No public URLs. All download access is via signed URLs generated server-side through a route that
first verifies the requesting actor has `admissions` or `admin` role for the tenant. Applicants
may only download their own documents (verified by application token).

Upload is via presigned upload URL generated by `POST /api/academy/admissions/applications/[id]/documents/upload-url`.
The route validates document type, file size (max 10 MB), and MIME type (PDF, JPEG, PNG only)
before issuing the URL.

### 4. Checklist enforcement

`canAdvanceToDecision(applicationId)` — service function that returns `true` only when every
`required` document type has a record with `status = received` or `status = waived`.

The admissions decision route calls `canAdvanceToDecision` before creating a decision record.
If the check fails, the route returns `422` with a safe message listing the missing documents.
Staff may override by waiving individual documents with a mandatory note.

### 5. Admin UI

- `/admin/admissions/[id]` — application detail page shows a checklist section: each document type,
  current status, upload date, and actions (mark received, waive with note, download).
- Staff can manually mark a document as received (for documents delivered outside the portal, e.g., mailed).
- Applicant view at `/apply/status` shows checklist status (pending/received/waived) for each required
  document and an upload button for pending items.

---

## Consequences

- Institutions can define faith-specific document types (pastoral reference, statement of faith, baptism record).
- Applications cannot advance to a decision decision if required documents are missing, reducing
  incomplete-file decisions.
- Storage costs scale with upload volume; private bucket and 10 MB limit mitigate runaway usage.
- Waiver audit trail (who waived, when, why) satisfies accreditation record-keeping requirements.

---

## Alternatives Considered

**Store documents as URLs pointing to external services (Google Drive, Dropbox):**
Rejected. No access control boundary. Links rot. Cannot enforce tenant isolation.

**Flat file references with no checklist enforcement:**
Rejected. Does not prevent staff from making decisions on incomplete applications.

**Required documents hardcoded in the codebase:**
Rejected. Faith-based institutions have diverse document requirements. Hardcoding creates a maintenance
burden and excludes institution-specific pastoral documents.

---

## Security / Privacy Review Notes

- Storage bucket is private. No public read access.
- Signed URLs expire in 15 minutes.
- Applicant access is scoped to their own application by token, not role.
- Document download route logs an audit event for every access.
- File type and size validation happens server-side before presigned URL issuance.
