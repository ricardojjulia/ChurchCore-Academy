# ADR-0039 — ShepherdAI Workflow Action Persistence

**Status:** Accepted
**Date:** 2026-06-22
**Deciders:** @ricardojjulia

---

## Context

ShepherdAI suggestions are generated by the evaluator and persisted in the `ai_suggestions` table via `ShepherdAiPostgresRepository`. ADR-0031 correctly decoupled suggestion reads from evaluator invocation.

However, staff actions on suggestions — promote, dismiss, snooze, and resolve — are written to `InMemoryAcademicWorkflowRepository`, which is reconstructed from scratch on every request. There is no Postgres-backed repository for these action events. Every action is lost on page reload.

This is a critical gap. ShepherdAI is the flagship differentiator of ChurchCore Academy's academic workflow surface. A workflow recommendation tool that loses all staff action state between page loads cannot function as a real work-tracking instrument. Staff cannot rely on it for handoffs, audit, or ongoing management.

The gap also blocks ShepherdAI signals from reaching downstream consumers: honor-roll computation, academic-standing checks, and the future ShepherdAI scheduling integration all depend on knowing whether a suggestion has already been acted upon.

---

## Decision

Replace `InMemoryAcademicWorkflowRepository` with a `ShepherdAiWorkflowActionRepository` backed by a new append-only `academy_shepherd_ai_workflow_actions` table.

**Table schema (migration):**

```sql
create table academy_shepherd_ai_workflow_actions (
  id          uuid primary key default gen_random_uuid(),
  tenant_id   text not null,
  suggestion_id uuid not null references ai_suggestions(id),
  action_type text not null check (action_type in ('promoted', 'dismissed', 'snoozed', 'resolved')),
  actor_id    text not null,
  note        text,
  snooze_until timestamptz,
  created_at  timestamptz not null default now()
);

create index on academy_shepherd_ai_workflow_actions (tenant_id, suggestion_id);
create index on academy_shepherd_ai_workflow_actions (tenant_id, actor_id);
```

**Repository contract:**

```ts
interface ShepherdAiWorkflowActionRepository {
  recordAction(tenantId: string, action: WorkflowActionInput, client: PoolClient): Promise<void>;
  getActionsForSuggestion(tenantId: string, suggestionId: string, client: PoolClient): Promise<WorkflowAction[]>;
  getLatestActionBySuggestion(tenantId: string, client: PoolClient): Promise<Map<string, WorkflowAction>>;
}
```

**Workflow queue page behavior:**

The queue page reads live `ai_suggestions` records and the latest persisted action per suggestion, then merges them into a computed display status:

- No action recorded → `open`
- Latest action is `snoozed` and `snooze_until` is in the future → `snoozed`
- Latest action is `promoted` → `in_progress`
- Latest action is `dismissed` or `resolved` → filter from default view (accessible via "All" filter)

This merge is computed at the read model layer, not stored as a mutable status column.

**Append-only contract:**

Action rows are never updated or deleted. If a suggestion is re-opened after dismissal, a new `promoted` action is appended. The full history is auditable.

**Tenant isolation:**

Every repository function accepts `tenantId` as the first argument and applies it as a WHERE clause before any other predicate. Cross-tenant access is structurally impossible at the query level.

---

## Consequences

**Positive:**
- Staff action state survives page reloads, tab switches, and session breaks.
- ShepherdAI becomes a usable operational work-tracking surface, not a demonstration.
- Full action history is auditable — who promoted, dismissed, or resolved each suggestion and when.
- Downstream consumers (honor roll, academic standing, scheduling) can read committed action state.
- The `InMemoryAcademicWorkflowRepository` is deleted — the in-memory path is not left as a fallback.

**Negative:**
- The action table grows without bound for institutions with high suggestion volume. A future archival policy will be needed.
- The computed display status adds a join and map-merge on every queue page load. This is acceptable at MVP scale but should be monitored.
- `snooze_until` semantics require the queue to re-surface snoozed suggestions on the next page load after the snooze expires — no background job is needed, but the page must filter correctly.

---

## Alternatives Considered

### Add a mutable `status` column to `ai_suggestions`

Rejected. `ai_suggestions` is evaluator output — the evaluator would overwrite staff action state on every re-evaluation run. Mutable status on the suggestions table conflates evaluator output with staff intent.

### Keep `InMemoryAcademicWorkflowRepository` and add localStorage persistence

Rejected. Browser storage is per-device, per-browser. Handoffs between staff members, mobile access, and multi-tab work would all see stale or divergent action state.

### Use a Redis cache for action state

Rejected. Academy persistence is Postgres. Introducing Redis for a single feature adds operational complexity and a new failure mode without a proportionate benefit at this scale.

---

## Review Notes

- **Security/privacy:** Action records contain `actor_id` (staff user ID). RLS must restrict reads to the same tenant. Actor identity comes from the verified Supabase session — never from a request header.
- **Testing:** Tests must cover: action recorded and retrieved, latest-action-per-suggestion map correctness, snooze expiry filtering, cross-tenant rejection, and empty-action-set behavior.
- **Migration:** Append-only. Never edit after commit.
- **ShepherdAI boundary:** Action persistence is infrastructure. ShepherdAI signal logic (which suggestions to generate) remains in the evaluator and must not be influenced by action history at signal-generation time.
