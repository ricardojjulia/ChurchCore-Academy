# Accepted Application Enrollment Conversion Design

Date: 2026-06-13
Status: proposed for implementation
Product area: admissions, people, enrollment, registration, audit

## Goal

Convert one accepted admission application exactly once into the minimum persistent SIS records required for a registrar-controlled student onboarding workflow.

## Scope

This slice creates:

- an active `student` role assignment for the accepted applicant;
- one Academy student profile with a tenant-unique student number;
- one program enrollment tied to the accepted application and program;
- one academic-period registration tied to the application term;
- immutable conversion events and global audit events;
- an idempotent registrar API and review action;
- a read-only conversion result in the admissions review surface.

This slice does not create:

- section or course registrations;
- invoices, tuition schedules, payment plans, or financial-aid records;
- LMS users, course shells, roster sync, or provider jobs;
- released Student PWA schedule, grades, documents, or messages;
- guardian relationships, advising assignments, or enrollment agreements;
- automatic conversion immediately when staff accepts an application.

## Users And Authority

Only same-tenant actors with `institution_admin`, `registrar`, or `admissions` may execute conversion.

Applicants, students, guardians, faculty, advisors, and deans may not execute conversion. A dean may review admissions decisions but student-record activation remains registrar-controlled.

The actor must come from the verified Academy session and request-scoped database context. Caller-provided person, tenant, role, student number, application status, or conversion state is never authoritative.

## Alternatives Considered

### Option A: Convert automatically when an application is accepted

Rejected. Acceptance and SIS activation have different operational authority and recovery requirements. Coupling them would make an admissions decision create academic identity and downstream obligations without a registrar checkpoint.

### Option B: Create only a student profile

Rejected. A profile without explicit program enrollment and academic-period registration is operationally ambiguous and cannot support registration, reporting, or later LMS synchronization reliably.

### Option C: Registrar-controlled transactional conversion

Selected. One explicit command creates the student role/profile, program enrollment, period registration, conversion event, and global audit event in one transaction. Retries return the original result.

## Domain Model

### Student profile

Reuse `academy_student_profiles` as the canonical student identity profile.

Conversion creates:

- `tenant_id`: application tenant;
- `person_id`: applicant person;
- `student_number`: generated by the database;
- `student_type`: `adult` for this first slice unless an institution-owned intake rule supplies another value in a later slice;
- `enrollment_status`: `active`;
- `program_id`: accepted application program;
- `guardian_required`: `false`;
- timestamps.

The existing applicant role remains as historical admissions authority. Conversion adds a separate active student role assignment; it does not rewrite or delete the applicant role.

### Program enrollment

Create `academy_program_enrollments` with:

- UUID primary key;
- tenant, student person, student profile, program, and source application references;
- status: `active`, `paused`, `completed`, `withdrawn`, or `cancelled`;
- admitted and enrolled timestamps;
- idempotency key and audit timestamps.

Only one enrollment may originate from a given application.

### Academic-period registration

Create `academy_period_registrations` with:

- UUID primary key;
- tenant, program enrollment, student person, and academic period references;
- status: `registered`, `cancelled`, `withdrawn`, or `completed`;
- registered timestamp;
- idempotency key and audit timestamps.

The accepted application must contain `application_term_id`. Conversion fails with `409` when no term is present. Selecting or changing the term belongs to an earlier admissions correction workflow, not to the conversion request.

### Conversion event

Create append-only `academy_enrollment_conversion_events` containing:

- tenant and application;
- actor;
- student profile, program enrollment, and period registration;
- event type `converted`;
- correlation ID and idempotency key;
- occurrence time.

The application receives immutable conversion metadata:

- `converted_at`;
- `converted_by_person_id`;
- `student_profile_id`;
- `program_enrollment_id`;
- `period_registration_id`.

Accepted applications with conversion metadata are considered converted. The application status remains `accepted`; conversion is not another admissions decision state.

## Student Number Allocation

Student numbers use a per-tenant sequence table:

- `academy_student_number_sequences(tenant_id, next_value, prefix)`;
- default prefix `S-`;
- allocation occurs under row lock inside the conversion transaction;
- output format is `<prefix><six-digit number>`, for example `S-000001`;
- the unique `(tenant_id, student_number)` constraint remains authoritative.

Clients may not provide or reserve student numbers.

## Transaction And Idempotency

The conversion API requires `Idempotency-Key`.

Within one request-owned database transaction:

1. lock and load the same-tenant application;
2. require status `accepted`;
3. authorize the actor;
4. return the existing conversion when the same idempotency key already completed it;
5. reject a different idempotency key when the application is already converted;
6. allocate a student number;
7. add the active student role if absent;
8. create the student profile;
9. create the program enrollment;
10. create the academic-period registration;
11. mark the application converted;
12. append the conversion event;
13. append the redacted global audit event;
14. commit.

Any failure rolls back every write.

## Database Security

All new tables:

- use composite tenant foreign keys;
- enable and force RLS;
- deny `anon`;
- allow same-tenant registrar, admissions, and institution-admin access;
- allow a student to read only their own program enrollment and period registrations;
- reject direct update or delete of conversion events;
- reject cross-tenant references at the constraint layer.

Database triggers enforce:

- only accepted applications may be converted;
- conversion metadata is all-or-none and immutable;
- one application cannot create multiple conversions;
- enrollment and registration identity fields are immutable.

## API

Add:

```text
POST /api/academy/admissions/applications/:id/convert
```

Request:

- verified session;
- `Idempotency-Key`;
- optional `X-Correlation-Id`;
- no JSON body.

Response:

```json
{
  "conversion": {
    "applicationId": "uuid",
    "studentProfileId": "uuid-or-text",
    "studentNumber": "S-000001",
    "programEnrollmentId": "uuid",
    "periodRegistrationId": "uuid",
    "status": "converted"
  }
}
```

Error mapping:

- `401`: no verified session;
- `403`: actor lacks conversion authority or tenant access;
- `404`: application not visible in the actor tenant;
- `409`: application not accepted, missing application term, already converted under another key, or concurrent conversion;
- `500`: generic response for unknown persistence failures.

Responses exclude event internals, audit metadata, database errors, role assignments, and provider data.

## Staff Surface

The admissions review page adds:

- conversion state;
- student number after conversion;
- a conversion action only for accepted, unconverted applications;
- explicit disabled messaging when the application has no term;
- no section-registration, billing, LMS, or Student PWA controls.

The action uses a client component only for the mutation button and result refresh. The page and queue remain server-rendered from persistent data.

## Recovery

- A failed transaction leaves the application unconverted.
- A network retry with the same idempotency key returns the original conversion.
- Incorrect program or term data must be corrected before conversion.
- Converted records are not deleted or rewritten through this workflow.
- Withdrawal, cancellation, transfer, duplicate-person resolution, and conversion reversal require later forward-event workflows.

## Testing

Required automated coverage:

- accepted application conversion succeeds once;
- submitted, declined, withdrawn, or termless applications conflict;
- unauthorized and cross-tenant actors are denied before writes;
- student numbers are tenant-scoped and concurrency-safe;
- duplicate retries return the same records;
- another idempotency key cannot reconvert the application;
- every foreign key is tenant-aware;
- all writes roll back when event or audit persistence fails;
- RLS role matrix covers unauthenticated, applicant, student-self, unrelated student, same-tenant registrar, and cross-tenant registrar;
- API responses expose only the conversion result;
- the admissions page renders eligible, blocked, and converted states;
- no LMS, billing, financial-aid, or section-registration side effect occurs.

## Delivery Gate

The slice is complete only when:

1. one accepted application produces exactly one student profile, program enrollment, and period registration;
2. every write is tenant-scoped, idempotent, auditable, and transactional;
3. database constraints and RLS block cross-tenant or unauthorized access;
4. the registrar can execute and review conversion from the persistent admissions surface;
5. the full test, lint, build, audit, migration, browser, and role-matrix gates pass;
6. documentation continues to state that section registration, billing, aid, Student PWA persistence, and LMS execution remain incomplete.
